================================================================================

  架构模式：前后端分离（B/S 架构）
  后端框架：Python 3.10+ / Flask 3.1

  核心算法：决策树分类 + 用户协同过滤

三、技术架构说明
================================================================================

————————————————————————————————————————
3.1  总体架构
————————————————————————————————————————

  本系统采用经典的前后端分离架构：

    用户浏览器
        ↓ HTTP 请求
    前端（Vue 3 + Vite，端口 3000）
        ↓ API 代理（/api → localhost:5000）
    后端（Flask，端口 5000）
        ↓ ORM 操作
    数据库（SQLite / MySQL）

  前端通过 Vite 的 proxy 配置将所有 /api 开头的请求代理转发到后端。
  后端接收请求后通过 Flask-SQLAlchemy 操作数据库，返回 JSON 数据。

————————————————————————————————————————
3.2  后端架构分层
————————————————————————————————————————

  后端采用三层架构模式：

  Routes 层（路由/控制器）
    ↓  接收 HTTP 请求，校验参数，调用 Service 层
  Services 层（业务逻辑）
    ↓  处理业务逻辑，调用推荐引擎、天气服务
  Models 层（数据模型）
    ↓  定义数据库表结构，提供 ORM 操作

————————————————————————————————————————
3.3  认证机制
————————————————————————————————————————

  认证流程：

  1) 用户登录 → 后端验证用户名密码
  2) 验证通过 → 后端用 Flask-JWT-Extended 生成 JWT Token
  3) Token 返回给前端 → 前端存储在 localStorage
  4) 后续请求 → 前端在请求头 Authorization 中携带 "Bearer <token>"
  5) 后端校验 Token → 识别用户身份

  Token 有效期：7天（在 config.py 中配置）
  Token 过期 → 后端返回 401 → 前端自动跳转登录页

————————————————————————————————————————
3.4  前端架构
————————————————————————————————————————

  前端采用 Vue 3 组合式 API 开发，核心技术栈：

  Vue 3          - 渐进式前端框架
  Vue Router     - 前端路由（SPA 单页应用导航）
  Pinia          - 状态管理（全局用户状态）
  Axios          - HTTP 请求库（与后端 API 通信）
  Element Plus   - UI 组件库（表格、表单、弹窗、图标等）
  ECharts        - 数据可视化（温度趋势图、统计图表）
  Day.js         - 日期时间处理

六、核心算法说明
================================================================================

————————————————————————————————————————
6.1  决策树分类算法（保暖等级预测）
————————————————————————————————————————

  文件位置：backend/app/services/recommendation_engine.py
  类名：    DecisionTreeRecommender

  功能：根据当前天气参数和出行场景，预测用户需要的保暖等级（1-5级）。

  输入特征（6维）：
    - temperature    当前温度（°C）
    - humidity       湿度（%）
    - wind_speed     风速（km/h）
    - precipitation  降水量（mm）
    - aqi            空气质量指数
    - scene_id       场景编号（0-6，对应7种场景）

  输出：warmth_level（1-5）
    1 = 非常凉爽（适合炎热天气）
    2 = 轻薄舒适
    3 = 适中
    4 = 偏暖
    5 = 最保暖（适合严寒天气）

  模型参数：
    - max_depth=8        最大深度8层
    - min_samples_split=3  最少3个样本才分裂
    - min_samples_leaf=2   叶子节点最少2个样本
    - random_state=42      固定随机种子保证可复现

  训练数据：内置35组标注样本，覆盖 -10°C 到 35°C 的不同天气和场景组合。

  个性化调整：
    - 怕冷体质(cold_sensitive)：保暖等级 +1
    - 怕热体质(heat_sensitive)：保暖等级 -1
    - 老年用户(age > 60)：保暖等级 +1

————————————————————————————————————————
6.2  协同过滤算法（个性化推荐优化）
————————————————————————————————————————

  文件位置：backend/app/services/recommendation_engine.py
  类名：    CollaborativeFilter

  功能：基于用户-物品偏好矩阵，利用余弦相似度找到品味相近的用户，
       参考他们的偏好对候选服装重新排序，实现个性化推荐。

  核心流程：

  1) 构建偏好矩阵
     从 user_preferences 表读取所有用户对服装的评分数据，
     构建 { user_id: { clothing_id: score } } 字典。

  2) 计算用户相似度
     对目标用户和其他用户，找到共同评分的服装项，
     用余弦相似度公式计算相似程度：

     sim(A,B) = (A·B) / (||A|| × ||B||)

     其中 A、B 分别是两个用户在共同物品上的评分向量。
     要求至少有2个共同评分项才计算。

  3) 推荐排序
     选取相似度最高的 top-5 用户作为"邻居"，
     对每件候选服装，加权计算邻居用户的评分：

     score(item) = Σ(sim_i × rating_i) / Σ|sim_i|

     按加权评分从高到低排序，选出最优推荐。

————————————————————————————————————————
6.3  推荐生成完整流程
————————————————————————————————————————

  文件位置：backend/app/services/recommendation_engine.py
  函数名：  generate_recommendation()

  完整流程：

  1) 接收参数：user_id、city、scene、weather_data

  2) 决策树预测保暖等级
     调用 dt_model.predict_warmth_level() 获得基础保暖等级

  3) 个性化体质调整
     根据用户体质类型和年龄微调保暖等级

  4) 候选服装筛选
     按5个槽位（上衣/下装/外套/鞋子/配饰）分别查询数据库，
     筛选条件：温度范围匹配、场景匹配、性别匹配、上架状态，
     按保暖等级差异排序取前20件候选。

  5) 协同过滤精排
     调用 cf_model.recommend_items() 对每个槽位的候选项做个性化排序，
     选出最匹配用户偏好的那一件。

  6) 动态穿搭决策
     根据保暖等级和天气条件决定是否需要外套和配饰：
     - 保暖等级 >= 3 或有降水或大风 → 需要外套
     - 保暖等级 >= 4 或有降水或重污染 → 需要配饰

  7) 生成穿衣建议文本
     根据温度、降水、风速、空气质量、紫外线等生成多条建议。

  8) 计算评分
     舒适度评分 = 80 - |temp-22|×2 - |humidity-55|×0.3 - wind×0.5
     搭配评分 = 舒适度 × 0.9 + 10

  9) 保存推荐记录到数据库并返回结果。

6.4  天气数据服务
————————————————————————————————————————

  文件位置：backend/app/services/weather_service.py

  天气数据来源：

  1) 和风天气 API（QWeather）
     配置文件：backend/app/config.py 中的 WEATHER_API_KEY
     API 地址：https://devapi.qweather.com/v7
     当配置了有效的 API Key 时，系统会调用真实天气接口获取数据。

  2) 本地气候模型（默认兜底方案）
     当未配置 API Key 或 API 调用失败时，系统使用内置的本地气候模型
     根据城市地理特征和月份生成模拟天气数据。该模型覆盖：
     北京、上海、广州、成都、杭州、哈尔滨、深圳等城市。

  缓存机制：
     天气数据按 (city, date) 缓存到 weather_records 表中，
     同一城市同一天不会重复请求 API，提高性能并节省 API 调用次数。

    数据库连接地址。
    开发环境默认: sqlite:///smart_outfit.db
    生产环境默认: mysql+pymysql://root:password@localhost/smart_outfit
    可通过环境变量 DATABASE_URL 覆盖。
